## Design Context

### Users
Three co-equal audiences: (1) frontend developers evaluating the library for integration — they want to understand the API, see it in action, and grab working code; (2) security engineers and architects assessing threat coverage, compliance posture, and audit capability; (3) enterprise technical decision-makers evaluating adoption risk — trust and credibility are the deciding factors. All three must feel served on the same page.

### Brand Personality
**Precise. Trustworthy. Sharp.**

Not theatrical. Not hacker-aesthetic. Not corporate-bland. The brand sits at the intersection of engineering excellence and security seriousness — like a well-maintained open-source project that happens to also be the most secure option. Think Stripe (precision, developer trust) or Linear (opinionated, sharp, never loud).

Emotional goal: a developer visiting should feel *competent* after a few minutes — like the library makes them smarter about security, not overwhelmed by it.

### Aesthetic Direction
**Light-first with controlled dark.** The current deep navy-black + emerald glow aesthetic is exactly what the user flagged as an anti-reference ("over-engineered dark UI: glassmorphism, neon glows, cyan-on-black"). Move toward a mostly-light design with decisive, tinted surfaces — not clinical white, but off-white with a subtle navy tint. Use a strong typographic hierarchy from a distinctive display font paired with a refined body font. Emerald green accent should feel like a precision mark, not a glow source.

Anti-references (explicitly rejected):
- Generic SaaS landing: purple-gradient hero + floating card grid
- Over-engineered dark UI: glassmorphism, neon glows, glow accents on black
- Corporate security vendor: padlock stock photo, blue/grey, no personality
- Developer-tool overload: too many tabs, panels, code blocks before value lands

### Design Principles

1. **Precision over decoration** — Every visual element must earn its place. No gradients for atmosphere, no glows for drama. Sharp edges, deliberate spacing, no flourishes that don't carry meaning.

2. **Interaction is the proof** — The primary CTA goal is "try a component now." Interactive demos belong above the fold or early in the scroll — not in section 7 of 8. Show the product working, don't only describe it.

3. **Hierarchy before quantity** — Eight sections with identical card grids creates noise. Fewer, more differentiated sections with varied layouts beat more content at uniform weight. Not everything deserves card treatment.

4. **Trust through restraint** — Security tools earn trust by not overselling. Confident, understated copy outperforms hype. The code comparison section is the strongest proof point — lead with it sooner.

5. **The audience is technical** — Don't explain what CSRF means. Don't pad copy with reassurance. Write like you're talking to a senior engineer who will read the source code before deciding.
